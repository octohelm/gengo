# 扩展自定义生成器规范

本文描述如何在 `gengo` 体系上扩展自定义生成器，以及扩展时需要遵守的接口、注册与兼容性约定。

## 适用范围

适用于以下任务：

- 新增自定义生成器
- 修改已有生成器实现
- 调整生成器注册方式
- 排查某个生成器为何没有被触发或输出异常

## 主入口

### 运行时接口

- `pkg/gengo/generator.go`
  - `Generator`：普通命名类型生成接口
  - `AliasGenerator`：类型别名生成接口
  - `GeneratorNewer`：按 `Context` 创建生成器实例
  - `GeneratorCreator`：自定义初始化流程
- `pkg/gengo/register.go`
  - 通过 `gengo.Register(...)` 注册生成器
- `pkg/gengo/context.go`
  - `Context.Doc(...)` 读取标签与注释
  - `Context.OptsOf(...)` 读取 `+gengo:<name>:...` 选项
  - `Context.Render(...)` / `RenderT(...)` 输出代码
  - `Context.Defer(...)` 在主生成后补写代码

### 参考实现

仓库内置生成器可作为扩展样例：

- `devpkg/deepcopygen`
- `devpkg/runtimedocgen`

## 扩展方式

### 注册约定

仓库内现有模式是通过 side import 触发 `init()` 注册：

1. 在独立包中实现生成器。
2. 在 `init()` 中调用 `gengo.Register(...)`。
3. 用 `doc.go` 说明该包通过 side import 生效。

除非任务明确要求改变入口组织，否则优先延续这一模式。

### 接口选择

- 只处理 `*types.Named`：实现 `Generator`
- 还要处理 `*types.Alias`：再实现 `AliasGenerator`
- 生成器需要绑定上下文状态：实现 `GeneratorNewer`
- 需要接入初始化钩子：实现 `GeneratorCreator`

### 触发规则

`pkg/gengo/context.go` 中的 `IsGeneratorEnabled` 语义如下：

- 存在 `+gengo:<name>` 即启用
- `+gengo:<name>=false` 显式禁用
- 只要存在 `+gengo:<name>:...` 选项标签，也视为启用

扩展生成器时应保持这一行为模型兼容，除非任务明确要求变更。

### 标签来源

对象上最终可见的标签由三层合并而来：

1. `GeneratorArgs.Globals`
2. 包级注释标签
3. 声明级注释标签

排查“某个生成器为什么被启用/禁用”时，先检查这三层。

## 输出约定

- 生成文件名由 `OutputFileBaseName` 与生成器名组成：`<base>.<generator>.go`
- 文件头固定写入 `// Code generated by gengo:<name> DO NOT EDIT.`
- 输出最终会经过 `pkg/format` 格式化
- `ErrSkip` 用于静默跳过当前声明
- `ErrIgnore` 用于停止当前生成路径但不报失败

扩展生成器时，不要随意破坏这些默认约定。

## 常见改动落点

### 新增或修改生成器逻辑

优先看：

- 目标生成器包本身
- `pkg/gengo/context.go`
- `pkg/gengo/generator.go`
- `pkg/gengo/genfile.go`

### 改标签解析或选项读取

优先看：

- `pkg/types/comments.go`
- `pkg/gengo/context.go`
- 相关生成器里对 `OptsOf(...)` 的使用

## 验证建议

- 只改某个生成器：`go test ./对应包/...`
- 改运行时接口或标签语义：至少覆盖 `./pkg/gengo/...` 与 `./pkg/types/...`
- 改动影响多个生成器：再考虑 `go test ./...`

如果本地无法完整验证，要说明未覆盖范围和原因。
